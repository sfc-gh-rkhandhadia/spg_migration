# MSSQL → Snowflake Postgres Migration Plugin

End-to-end migration toolkit: DDL realization, data load, and two-sided parity validation.

---

## Skills

| Skill | Purpose |
|---|---|
| `mssql-ddl-realization` | Deploy customer DDL into a disposable SQL Server instance; generate a semantic witness dataset so every view and procedure returns rows |
| `mssql-spg-load` | Load data from MSSQL into Snowflake Postgres (SPG) |
| `mssql_spg_migration_validation_testing` | Run two-sided parity testing between SQL Server and SPG; produce sign-off report |
| `mssql-spg-orchestrator` | Coordinate all three skills end-to-end from a single prompt |

---

## Required Inputs for a Fresh Run

| Input | What it is |
|---|---|
| **DDL folder** | Flat directory of `*.sql` files exported from SSMS — one object per file, `GO`-delimited |
| **SPG host** | Snowflake Postgres hostname (e.g. `abc123.us-east-1.aws.postgres.snowflake.app`) |
| **SPG user / password / database** | Credentials for the target SPG instance |
| **Shared output directory** | Any writable local path — all intermediate files land here |

Nothing else is required. Every file in the shared folder is derived from these four inputs.

## Fresh Run — Step Order

Steps 1 and 2 have no dependency on each other and can run in parallel.
Steps 3–9 are strictly sequential — each step requires its predecessor's output.

```
1. discover_spg_constraints.py   ← needs SPG credentials only
2. parse_ddl.py                  ← needs DDL folder only
3. build_dep_graph.py            ← needs outputs from 1 + 2
4. deploy_schema.py              ← needs outputs from 2 + 3  +  MSSQL (Docker)
5. seed_data.py                  ← needs outputs from 2 + 3 + 4  +  MSSQL
6. validate_chains.py            ← needs outputs from 2 + 3 + 4 + 5  +  MSSQL
7. generate seed_profiles.yaml   ← needs output from 6
8. loader (load_mssql_to_spg)    ← needs MSSQL + SPG credentials  +  output from 2
9. validation scripts            ← needs MSSQL + SPG  +  seed_profiles.yaml
```

### Key constraint

`object_inventory.json` is the spine — every downstream step depends on it. If the DDL folder changes, re-run `parse_ddl.py` first and regenerate all subsequent files in order. No file from a prior run against a different DDL set can be reused.

## Invoking end-to-end via the orchestrator

```
/mssql-spg-orchestrator

DDL folder:   /path/to/customer/ddl/
SPG host:     <spg-host>.aws.postgres.snowflake.app
SPG user:     snowflake_admin
SPG database: <database>
Output:       /path/to/validation_results/
```

The orchestrator runs all four skills in dependency order and assembles a final `run_manifest.json` summary.

## Script Locations

```
mssql-ddl-realization/scripts/
  parse_ddl.py                  — parse DDL folder → object_inventory.json
  build_dep_graph.py            — build dependency graph → dep_graph.json
  deploy_schema.py              — deploy schema to MSSQL
  seed_data.py                  — generate witness dataset
  validate_chains.py            — confirm views/functions return rows
  adaptive_seed.py              — join consistency fix (library, not CLI)

mssql-spg-load/scripts/
  discover_spg_constraints.py   — discover SPG CHECK constraints → spg_column_constraints.json
  load_mssql_to_spg.py          — copy data from MSSQL to SPG

mssql_spg_migration_validation_testing/scripts/
  validate_batch.py             — run view parity tests
  mssql_proc_executor.py        — execute procedures on MSSQL side
  spg_proc_executor.py          — execute procedures on SPG side
  compare_proc_outputs.py       — diff + classify results
  full_validation.py            — end-to-end validation runner
  generate_validation_markdown.py  — produce markdown sign-off report
  generate_migration_report.py     — produce Snowflake-branded PowerPoint deck
```
